# v0.3 → v0.4 slug migration plan

The `ailiance-agent` fork rebrands the marketing surface (extension
display name, walkthrough id, marketplace categories, narrowed CSP)
in v0.3, but **defers** the internal `dirac.*` slug rename to v0.4.

This document records what is still using the upstream `dirac` slug,
why it was deferred, and the migration steps to take when v0.4 lands.

## What we rebranded in v0.3

- `package.json` walkthroughs[0].id: `IsaacWalkthrough` →
  `AgentKikiWalkthrough` (with matching call sites in
  `src/extension.ts` and `src/core/controller/ui/openWalkthrough.ts`).
- CSP whitelist in `src/core/webview/WebviewProvider.ts`: dropped
  `https://*.dirac.run` from `connect-src` (telemetry is off in this
  fork, so the host is unused).
- (Already done in v0.1) `displayName`, `categories`, `keywords`.

## What still uses the `dirac` slug (deferred to v0.4)

There are ~660 occurrences of `dirac.*` in the source tree. The
high-traffic categories:

| Surface | Example | Why deferred |
| --- | --- | --- |
| VS Code command ids | `dirac.openWalkthrough`, `dirac.plusButton`, `dirac.popoutButton`, `dirac.focusChatInput`, ~40 commands total | Renaming would break every `vscode.commands.executeCommand("dirac.X")` callsite **and** any user keybinding/macro referencing the old id. Must be done with a `commands` alias migration shim and one release of overlap. |
| Configuration namespace | `dirac.*` settings in `package.json#contributes.configuration` | Persisted in user `settings.json`. A naive rename loses every existing user setting. Needs a v0.4 migrator that reads `dirac.*`, writes `ailiance-agent.*`, then removes the old keys. |
| View container ids | `claude-dev-ActivityBar`, `dirac-sidebar` | Already partially branded; the remaining `dirac-*` ids are wired into VS Code state and would lose pinned positions on rename. |
| Proto packages | `@shared/proto/dirac/*` | TypeScript-internal; pure rename, but ~50 files of churn. Schedule for the v0.4 proto regeneration cycle. |
| Storage paths | `~/.dirac/`, `.dirac/` | User data location. Needs a copy-on-first-launch migration. |

## Migration helper sketch (v0.4)

```typescript
// 1. settings migrator (run on extension activate, exactly once)
const cfg = vscode.workspace.getConfiguration()
for (const key of cfg.inspect("dirac")?.globalValue ? Object.keys(...) : []) {
    const oldVal = cfg.get(`dirac.${key}`)
    if (oldVal !== undefined && cfg.get(`ailiance-agent.${key}`) === undefined) {
        await cfg.update(`ailiance-agent.${key}`, oldVal, vscode.ConfigurationTarget.Global)
    }
}

// 2. command alias shim (register both old + new ids for one release)
context.subscriptions.push(
    vscode.commands.registerCommand("dirac.openWalkthrough", () =>
        vscode.commands.executeCommand("ailiance-agent.openWalkthrough"),
    ),
)

// 3. storage migrator (copy ~/.dirac/* → ~/.ailiance-agent/* with marker file)
```

## Why we did NOT do this in v0.3

The fork's first goal is **shipping a working binary**. Renaming
~660 occurrences across command ids, config keys, view containers,
proto schemas, and persisted state — without breaking
`pnpm package`, the published extension, OR existing users' settings
— is a release of its own. v0.3 is the minimum viable rebrand
(marketing + walkthrough + CSP). v0.4 is the deep rename behind a
proper migration shim with one release of overlap.
